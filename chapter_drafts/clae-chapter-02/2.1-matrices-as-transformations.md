# Chapter 2: Matrices and Linear Transformations

In the Introduction we did something quiet but consequential: we agreed to treat
data as geometry. A house in Ames stopped being a row in a spreadsheet and became
a point in a space, and a dataset became a cloud of such points. That agreement is
the foundation of everything we will do in this book. But a cloud of points just
sits there, and estimation does not sit still. Estimation acts. It projects a noisy
measurement onto the directions that carry signal, it rotates a cloud to line up
with its own axes, it collapses many correlated features down to a few. The object
that does the acting is the matrix.

You have met the matrix before. Almost certainly it was handed to you as a
rectangular grid of numbers, together with rules for adding and multiplying such
grids. That description is correct, and for our purposes it is nearly useless: it
tells us what a matrix is made of and says nothing about what a matrix does. In
this book we think of a matrix differently. A matrix is a **transformation**. It
takes a vector and returns a vector. Least squares is a matrix acting on data.
Principal component analysis is a matrix acting on data. The Kalman filter is a
matrix acting on data. Learn to see the action and the rest of the book opens up.

By the end of this chapter we will be able to read a matrix as a transformation and
predict what it does through the column picture. We will compute a matrix-vector
product two ways and see what each way reveals that the other hides. We will meet
the handful of geometric transformations, projection above all, that come back
again and again in estimation. We will ask when the system $A\mathbf{x} =
\mathbf{b}$ can be solved, and we will read the answer as a statement about the
data. And we will standardize the Ames housing features as a single linear
transformation, leaving them ready for the covariance and principal components of
Part II. Each of these is one more way to reach into a cloud of noisy measurements
and pull out the structure inside.

## 2.1 Matrices as transformations

Let us start with the operation itself and read it slowly. Take a vector
$\mathbf{x}$ with $n$ entries and a matrix $A$ with $m$ rows and $n$ columns. The
product $A\mathbf{x}$ is a vector with $m$ entries. Nothing here is new. What is new
is how we read it. We read $A\mathbf{x}$ as a function: $A$ accepts an input vector
and returns an output vector, and we give that function a name by writing
$T(\mathbf{x}) = A\mathbf{x}$. The matrix $A$ is the rule, and $T$ is the
**transformation** that rule carries out. An $m \times n$ matrix sends vectors from
$\mathbb{R}^n$ to vectors in $\mathbb{R}^m$. When $m = n$ the transformation sends a
space to itself, which is the case for most of the operations we will care about.

### The simplest transformation is a difference

Consider the matrix

$$A = \begin{bmatrix} 1 & 0 & 0 \\ -1 & 1 & 0 \\ 0 & -1 & 1 \end{bmatrix}.$$

Apply it to a vector $\mathbf{x} = (x_1, x_2, x_3)$ and read off the result:

$$A\mathbf{x} = \begin{bmatrix} x_1 \\ x_2 - x_1 \\ x_3 - x_2 \end{bmatrix}.$$

The transformation replaces each entry with its difference from the one before it.
This $A$ is a **difference matrix**, and it is a genuine action on data. Hand it a
running total and it hands back the increments. The arithmetic is elementary, and
that is exactly the point: we have stopped asking what the numbers in $A$ are and
started asking what $A$ does.

### Linearity

The transformations a matrix performs are not arbitrary. They are **linear**, which
means they respect addition and scaling. For any vectors $\mathbf{x}$ and
$\mathbf{y}$ and any scalars $a$ and $b$,

$$T(a\mathbf{x} + b\mathbf{y}) = a\,T(\mathbf{x}) + b\,T(\mathbf{y}).$$

Matrix multiplication obeys this identity exactly, which is why every matrix is a
linear transformation, and, it turns out, every linear transformation between
finite-dimensional spaces is some matrix. Linearity has a geometric face worth
holding onto. A linear transformation fixes the origin, sends straight lines to
straight lines, and keeps evenly spaced points evenly spaced. It can rotate the
space, stretch it, flatten it, or shear it. It cannot bend it. That rigidity is
exactly what makes linear estimation tractable: to know what a transformation does
to a whole dataset, we only need to know what it does to a few directions.

### The column picture

There is one way of reading $A\mathbf{x}$ that will carry us through the entire
book, so it is worth slowing down for it. Write $A$ out by its columns,
$\mathbf{a}_1, \mathbf{a}_2, \ldots, \mathbf{a}_n$. Then

$$A\mathbf{x} = x_1 \mathbf{a}_1 + x_2 \mathbf{a}_2 + \cdots + x_n \mathbf{a}_n.$$

The output is a **linear combination of the columns of $A$**, and the entries of
$\mathbf{x}$ are the amounts of each column we take. This is the **column picture**,
and it changes how we see the product. The input $\mathbf{x}$ is no longer data
being acted on. It is a recipe, a list of how much of each column to add. Whatever
outputs we can reach are exactly the combinations the columns can build, no more
and no less. We will come back to this picture every time it matters: when we ask
whether $A\mathbf{x} = \mathbf{b}$ has a solution in Section 2.4, when we project
onto the space the columns span in Chapter 11, and when we hunt for the directions
a dataset actually occupies in Chapter 10.

Return to the difference matrix and read it through the column picture. Its columns
are $(1, -1, 0)$, $(0, 1, -1)$, and $(0, 0, 1)$, so $A\mathbf{x}$ is the combination
$x_1(1, -1, 0) + x_2(0, 1, -1) + x_3(0, 0, 1)$. Add those weighted columns and we
recover $(x_1, x_2 - x_1, x_3 - x_2)$, exactly what the row-by-row arithmetic gave
us. It is the same product seen from two sides. Row by row it is a list of
differences. Column by column it is a recipe over three fixed directions. Figure
2.1 draws a single input vector and its image under $A$, with the transformation
shown as the arrow that carries one point to the other.

Reading a matrix as an action raises a fair question. How do we actually compute the
action, and does the column picture compete with the row-by-row arithmetic most of
us learned first? It does not. They are the same product seen from two sides, and
each tells us something the other keeps quiet. In Section 2.2 we set the two
readings side by side, and then we compose transformations, building one matrix that
performs two actions in sequence.
